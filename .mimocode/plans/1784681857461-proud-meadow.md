# Plano de Deploy — MuMarket (MuDream Market Tracker)

## Objetivo
Publicar o site MuMarket para funcionar online com monitoramento ativo.

## Arquitetura de Deploy
- **Frontend**: Vercel (React + Vite, estático)
- **Backend**: Railway (Node.js + Express + Socket.IO, Docker)
- **Domínio**: `mumarket.jdwebstudio.com.br`
- **Banco de dados**: Supabase (já configurado)

---

## Passo 1: Preparar o Backend para Deploy

### 1.1 Verificar e corrigir configs de produção
- Arquivo: `backend/src/config.ts` — confirmar que lê `PORT` do env (default 3000)
- Arquivo: `backend/src/index.ts` — confirmar que serve `frontend/dist/` em produção com SPA fallback
- Arquivo: `backend/src/services/market-monitor.ts` — confirmar que funciona sem Redis (Redis é opcional)

### 1.2 Verificar Dockerfile
- Arquivo: `Dockerfile` — já está correto (node:22-slim, npm install, build backend, CMD node dist/src/index.js)
- Railway usará este Dockerfile via `railway.json`

### 1.3 Variáveis de Ambiente necessárias (Railway)
```
PORT=3000
SUPABASE_URL=https://rlkwezadhrnskzgdzlpa.supabase.co
SUPABASE_ANON_KEY=<valor do .env>
SUPABASE_SERVICE_ROLE_KEY=<valor do .env>
JWT_SECRET=<gerar chave segura>
JWT_REFRESH_SECRET=<gerar chave segura>
CORS_ORIGIN=https://mumarket.jdwebstudio.com.br
MUDREAM_GRAPHQL_ENDPOINT=https://mudream.online/api/graphql
MUDREAM_COOKIE=<cookie do MuDream - configurar depois>
```

## Passo 2: Deploy Backend no Railway

### 2.1 Criar conta e projeto no Railway
- Acessar https://railway.app
- Criar conta (free tier: 500h/mês)
- "New Project" → "Deploy from GitHub repo"
- Conectar repositório GitHub

### 2.2 Configurar build
- **Root Directory**: `/` (raiz do monorepo)
- O `railway.json` já define: Dockerfile builder, start command `node dist/src/index.js`

### 2.3 Adicionar variáveis de ambiente
- Copiar valores de `backend/.env` e `backend/.env.production`
- Definir `CORS_ORIGIN=https://mumarket.jdwebstudio.com.br`

### 2.4 Verificar deploy
- Railway atribui URL como `seu-app.up.railway.app`
- Testar: `GET /api/health` deve retornar `{ status: 'ok', timestamp: ... }`

---

## Passo 3: Deploy Frontend no Vercel

### 3.1 Criar conta e projeto no Vercel
- Acessar https://vercel.com
- Conectar conta GitHub
- "New Project" → Import Git Repository → selecionar MuMarket

### 3.2 Configurar build
- O `vercel.json` já define:
  - `buildCommand`: `cd frontend && npm install && npm run build`
  - `outputDirectory`: `frontend/dist`
  - `framework`: `vite`

### 3.3 Variáveis de Ambiente
```
VITE_API_URL=https://seu-app.up.railway.app
```
(Usar a URL real do Railway)

### 3.4 Deploy
- Vercel faz deploy automático a cada push
- URL padrão: `https://mumarket.vercel.app`

---

## Passo 4: Configurar Domínio Customizado

### 4.1 No Vercel
- Settings → Domains → Adicionar `mumarket.jdwebstudio.com.br`
- Vercel mostra registros DNS necessários

### 4.2 No Hostinger (ou onde estiver o DNS)
- DNS Management → Adicionar registros:
  - **CNAME**: `mumarket` → `cname.vercel-dns.com`
  - ou **A** records conforme instruções do Vercel

### 4.3 SSL
- Vercel configura SSL automaticamente
- Aguardar propagação do DNS (até 24h)

---

## Passo 5: Configurar CORS e URLs

### 5.1 Railway (Backend)
- Atualizar `CORS_ORIGIN` para `https://mumarket.jdwebstudio.com.br`
- Railway faz redeploy automático

### 5.2 Vercel (Frontend)
- `VITE_API_URL` apontando para URL do Railway

---

## Passo 6: Verificar Monitoramento

### 6.1 Testar fluxo completo
1. Acessar `https://mumarket.jdwebstudio.com.br`
2. Login deve funcionar (JWT via Supabase)
3. Socket.IO deve conectar (verificar indicador verde no dashboard)
4. Criar um filtro de teste
5. Configurar cookie do MuDream nas Settings
6. Iniciar monitoramento no Dashboard
7. Verificar que alertas aparecem quando itens combinam com filtros

### 6.2 Verificar health check
- `GET /api/health` → `{ status: 'ok' }`
- `GET /api/monitoring/status` → status do monitoramento

## Notas

- O cookie do MuDream (`MUDREAM_COOKIE`) precisa ser configurado manualmente após o deploy (via Settings page)
- Redis é opcional — o sistema funciona sem ele
- Puppeteer/CDP não funciona em cloud — apenas modo HTTP (que é o fallback)
- Backend roda na porta 3000 em produção (via env PORT)
- Frontend em produção é servido como arquivos estáticos pelo backend na raiz `/` (quando acessado via domínio do backend) OU pelo Vercel (quando acessado via domínio do frontend)
